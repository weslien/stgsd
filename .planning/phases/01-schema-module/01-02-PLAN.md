**Lifecycle hooks:**

**Project reducers (SCHM-01):**
- `insert_project`: params (git_remote_url, name, description, core_value, constraints, context, key_decisions), inserts with `id: 0n`, `created_at: ctx.timestamp`, `updated_at: ctx.timestamp`
- `update_project`: params (project_id: t.u64(), name, description, core_value, constraints, context, key_decisions), finds existing by PK, throws SenderError if not found, spreads existing + overrides + `updated_at: ctx.timestamp`
- `delete_project`: params (project_id: t.u64()), finds existing, throws SenderError if not found, deletes by PK

**Phase reducers (SCHM-02):**
- `insert_phase`: params (project_id: t.u64(), number, name, slug, goal, status, depends_on, success_criteria). Referential integrity: `ctx.db.project.id.find(project_id)` -- throw SenderError if project not found. Insert with `id: 0n`, timestamps.
- `update_phase`: params (phase_id: t.u64(), number, name, slug, goal, status, depends_on, success_criteria). Find existing, throw if not found, spread-update.
- `delete_phase`: params (phase_id: t.u64()). **CASCADE DELETE** -- find phase, then delete all children in order:
  1. For each plan in `ctx.db.plan.plan_phase_id.filter(phase_id)`:
     a. For each task in `ctx.db.planTask.plan_task_plan_id.filter(plan.id)`: delete task
     b. For each summary in `ctx.db.planSummary.plan_summary_plan_id.filter(plan.id)`: delete summary
     c. For each mustHave in `ctx.db.mustHave.must_have_plan_id.filter(plan.id)`: delete mustHave
     d. Delete plan
  2. For each verification in `ctx.db.verification.verification_phase_id.filter(phase_id)`: delete
  3. For each research in `ctx.db.research.research_phase_id.filter(phase_id)`: delete
  4. For each phaseContext in `ctx.db.phaseContext.phase_context_phase_id.filter(phase_id)`: delete
  5. For each requirement in `ctx.db.requirement.requirement_project_id.filter(phase.project_id)` where requirement.phase_number matches phase.number: delete
  6. Delete phase itself

  IMPORTANT: When iterating index filters for deletion, collect items into an array first (spread the iterator: `[...ctx.db.plan.plan_phase_id.filter(phase_id)]`), THEN delete. Do not delete while iterating.

**Plan reducers (SCHM-03):**
- `insert_plan`: params (phase_id: t.u64(), plan_number: t.u64(), type, wave: t.u64(), depends_on, objective, autonomous: t.bool(), requirements, status, content). Referential integrity: check phase exists. Insert with `id: 0n`, timestamps.
- `update_plan`: params (plan_id: t.u64(), plan_number: t.u64(), type, wave: t.u64(), depends_on, objective, autonomous: t.bool(), requirements, status, content). Spread-update.
- `delete_plan`: params (plan_id: t.u64()). Delete plan's tasks, summaries, and must-haves first, then delete plan.

**PlanTask reducers (SCHM-04):**
- `insert_plan_task`: params (plan_id: t.u64(), task_number: t.u64(), type, description, status, commit_hash as t.string() -- use empty string for no hash). Referential integrity: check plan exists. Insert with `id: 0n`, timestamps. Note: commit_hash is optional in the table but reducers accept t.string() -- pass empty string to mean "none" or use the value.
- `update_plan_task`: params (task_id: t.u64(), task_number: t.u64(), type, description, status, commit_hash). Spread-update.
- `delete_plan_task`: params (task_id: t.u64()). Simple delete.

**Requirement reducers (SCHM-05):**
- `insert_requirement`: params (project_id: t.u64(), category, number, description, status, phase_number, milestone_version). Referential integrity: check project exists. Insert with `id: 0n`, timestamps. For milestone_version, accept t.string() -- empty string means "none".
- `update_requirement`: params (requirement_id: t.u64(), category, number, description, status, phase_number, milestone_version). Spread-update.
- `delete_requirement`: params (requirement_id: t.u64()). Simple delete.

**Key patterns to follow throughout:**
- Always use `0n` for auto-inc ID on insert
- Always use `ctx.timestamp` for timestamps
- Always spread existing row on update: `{ ...existing, ...fields, updated_at: ctx.timestamp }`
- Always check parent exists before insert (referential integrity)
- Always throw `new SenderError('...')` with descriptive message on validation failure
- For optional fields in the table (t.string().optional()), the reducer can accept t.string() and pass empty string or null as needed. Use the t.string() param type and handle the "optional" semantics by passing null when appropriate. Actually, since SpacetimeDB reducers require typed params, accept all as t.string() and the caller passes empty string for "no value". In the insert, map empty string to undefined/null for optional columns if the SDK supports it, or simply pass the empty string -- the optional column can hold any string value.

**seed_project reducer (bulk initialization):**
- `seed_project`: params (git_remote_url: t.string(), name: t.string(), description: t.string(), core_value: t.string(), constraints: t.string(), context: t.string(), key_decisions: t.string(), phases_json: t.string(), requirements_json: t.string())
- Logic:
  1. Insert project row (id: 0n, timestamps) -- capture returned row for `proj.id`
  2. Parse `phases_json` with `JSON.parse()` -- expects array of `{ number, name, slug, goal, status, depends_on, success_criteria }`
  3. For each phase: insert with `project_id: proj.id`, capture returned row for phase IDs
  4. Parse `requirements_json` with `JSON.parse()` -- expects array of `{ category, number, description, status, phase_number, milestone_version }`
  5. For each requirement: insert with `project_id: proj.id`
  6. Insert initial project_state row with `project_id: proj.id`, current_phase: first phase number, current_plan: 0n, current_task: 0n, last_activity: ctx.timestamp, empty strings for other text fields
  7. If any JSON.parse fails, throw SenderError with descriptive message

Note on seed_project parameter design: Using JSON strings for phases/requirements avoids deeply nested `t.object()` / `t.array()` type definitions. JSON.parse is deterministic (pure function on its input) and safe in reducers.
